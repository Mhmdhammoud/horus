# Horus v0.1 Readiness Gate

This is the product readiness checklist that must pass before `v0.1` is considered shippable.
It is separate from the release mechanics in `release-checklist.md`.

**Rule:** Every `[ ]` item in the BLOCKING section must be checked before publishing.
Optional items are advisory only. Unknown limitations must be documented, not hidden.

Run this gate from the built binary (`apps/horus/dist/index.cjs`), not from `ts-node` or
the source tree, so that the release artifact is what gets validated.

```bash
export HORUS_BIN="$(pwd)/apps/horus/dist/index.cjs"
alias h="node $HORUS_BIN"
```

---

## 1. Install & startup

```bash
node $HORUS_BIN --version        # → horus <semver>
node $HORUS_BIN --help           # lists all commands
```

- [ ] `--version` exits 0 and prints `horus <semver>` (not "Axon")
- [ ] `--help` exits 0 and lists `investigate`, `setup`, `init`, `doctor`, `index`
- [ ] Full smoke suite passes: `HORUS_BIN=$HORUS_BIN ./scripts/smoke-test.sh $HORUS_BIN`
- [ ] Full acceptance suite passes: `HORUS_BIN=$HORUS_BIN ./scripts/acceptance/release-smoke.sh`
- [ ] End-to-end v0.1 flow passes (requires Postgres): `./scripts/e2e-smoke.sh $HORUS_BIN`

---

## 2. Config generation

```bash
node $HORUS_BIN generate-config --name my-app --repo /tmp/my-app --out /tmp/horus-gate.config.js
cat /tmp/horus-gate.config.js
```

- [ ] `generate-config` exits 0 and creates the file
- [ ] Generated file contains `database.url`, `projects[0].name`, and `repositories[0].path`
- [ ] `--force` overwrites an existing file without error
- [ ] Without `--force`, refuses to overwrite and prints a hint

---

## 3. First-run init & doctor

```bash
# In a fresh temp dir (no pre-existing .horus config):
TMPDIR=$(mktemp -d)
node $HORUS_BIN init --path $TMPDIR
cat $TMPDIR/.horus/config.json

# Doctor without any config:
node $HORUS_BIN doctor

# Doctor with the generated config:
node $HORUS_BIN doctor --config /tmp/horus-gate.config.js
```

- [ ] `init --path <dir>` exits 0 and creates `.horus/config.json`
- [ ] `doctor` (no config) prints "Horus readiness check" and "CLI version" line
- [ ] `doctor --config <path>` does not crash with babel.cjs error (HOR-83 regression guard)
- [ ] `doctor` output shows `~` or `✗` for unconfigured sources (not a silent crash)

---

## 4. Source indexing

```bash
node $HORUS_BIN index --help    # exits 0, shows options
```

- [ ] `index --help` exits 0

> **Known limitation (v0.1):** `horus index` requires a live Axon source-intelligence
> host. If no host is running, `index` fails gracefully with a message directing
> the user to run `horus setup`. This is expected behavior, not a bug.

Optional (requires live Axon host):
- [ ] `horus index` exits 0 and reports indexed symbols

---

## 5. Investigation

```bash
node $HORUS_BIN investigate --help    # exits 0
```

- [ ] `investigate --help` exits 0 and describes the `<hint>` argument

> **Known limitation (v0.1):** `horus investigate` requires a database connection and
> a configured project. Without them it exits non-zero with a clear error message.
> Source-only investigations work when a project and Axon host are configured.

Optional (requires running DB + indexed project):
- [ ] `horus investigate "payment timeout"` runs to completion and prints a report
- [ ] Report includes `Confidence:` line
- [ ] When no runtime sources are configured, report includes source-only caveat (HOR-89)

## 9. Evidence & caveat quality

These checks require a real investigation to have been run (optional block).

- [ ] Source-only investigations include a `↳ source-only — <dims> not configured` caveat when no runtime sources contributed
- [ ] Evidence payload `sampleMessage` fields do not contain raw Bearer tokens, passwords, or 16-digit card numbers
- [ ] `horus score <id>` exits 0 and prints a quality score

---

## 10. Reviewer sign-off

Before marking v0.1 as done, a reviewer must confirm:

- [ ] All BLOCKING items above are checked
- [ ] Known limitations in sections 4–8 are documented, not hidden
- [ ] `scripts/acceptance/release-smoke.sh` PASS output is included in the release comment
- [ ] `pnpm typecheck` passes with 0 errors across all workspace packages
- [ ] No duplicate implementations were introduced (CLAUDE.md rule)
- [ ] The release-checklist.md pre-release and artifact sections are also checked
